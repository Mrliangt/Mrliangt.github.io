Agent Skill 设计详解：fullchain-block-audit

一、Skill 是什么？——来自你的 PDF 的核心定义

你的 Skill Engineering 设计规范.pdf 给出了一个非常精准的定义：

Skill 把”每次都需要人来指挥的重复性任务”封装成 Agent 可以自主执行、可复现、可验证的流程。

关键词是不是 Prompt。一个 Prompt 是”你帮我做 X”，而一个 Skill 是”给 Agent 足够的信息，让它自己完成 X，不需要人追问”。

PDF 中还给出了 Skill 的运行时模型：

一个合格的 Skill 必须具备 7 个要素：输入定义、状态管理、工具编排、决策机制、错误恢复、验证机制、输出契约。缺了就退化成了”用户 → Prompt → 模型 → 输出”。

二、8 条设计原则逐条对照

你的 Skill Engineering 设计规范.pdf 提出了 8 条设计原则。下面逐条分析 fullchain-block-audit 是如何遵循的：

原则 1：架构设计——Skill 不是 Prompt

要素 fullchain-block-audit 的做法
输入定义 用户提供一个”屏蔽需求”（谁被屏蔽、屏蔽什么内容、哪些渠道）
状态管理 10 步 Audit Checklist 追踪当前执行到哪一步
工具编排 指向具体的代码文件路径（block-upuser-strategy/ 目录）
决策机制 Channel Inventory 表格 + New Channel Onboarding SOP
错误恢复 “Critical Design Decisions” 部分覆盖了 Redis 故障、forward_info 为空等异常
验证机制 Monitoring 表格定义了 4 个告警指标和阈值
输出契约 最终交付物是”所有渠道都有屏蔽覆盖 + 新渠道接入 SOP”

不足之处：当前版本的状态管理和错误恢复还比较弱——只有 checklist 和告警阈值，没有具体的”验证失败后怎么回退”的流程。如果要做得更完善，可以加一个验证脚本。

原则 2：模型无关性——“高档位模型开发，低档位模型执行”

这条原则说的是：如果把底层模型从 Claude Opus 换成 Haiku，Skill 还能不能跑？ 如果能，说明流程足够solid；如果不能，说明过度依赖模型的推理能力。

fullchain-block-audit 的设计：

  • Audit Checklist 是确定性的步骤列表，不需要模型推理
  • Channel Inventory 是硬编码的表格，不需要模型猜测
  • Implementation Pattern 给出了具体的代码模板，不需要模型自己设计
  • New Channel Onboarding SOP 是5 步确定性流程

模型需要做的判断只有一处：”这个新渠道的资源类型是什么？用哪个提取器？” 这个判断是真正需要 LLM 推理的部分，其余都是”照着做”。

结论：低档位模型应该能执行这个 Skill，因为它的大部分工作是”查表 → 套模板 → 注册”。

原则 3：关注点分离——确定性计算给脚本，探索性推理给模型

任务类型 应该由谁做 fullchain-block-audit 的处理
检查哪些 Lua 文件注册了过滤算子 脚本（grep） 目前由模型做 → 可改进
判断新资源类型的 UP 主 ID 字段名 模型（需要看代码理解语义） 由模型做 → 正确
生成提取器代码 模型（套模板） 由模型做 → 正确
验证编译是否通过 脚本（bash build.sh） 目前未包含 → 可改进

PDF 说：”把所有能脚本化的东西都脚本化，只把真正需要判断的留给模型。”

改进方向：可以加一个 scripts/audit_channels.sh 脚本，自动扫描所有 Lua 配置文件，检查是否包含 BlockListLoadStrategy 和 BlockUpuserFilterStrategy，输出缺失的渠道列表。

原则 4：调用边界——Skill 之间调能力接口，不调内部脚本

当前只有一个 Skill，没有跨 Skill 调用的场景。但如果未来扩展（比如需要一个 log-analysis Skill 来分析屏蔽过滤的日志），应该通过能力接口调用，不能直接读对方的脚本。

原则 5：原子化与编排——不要写万能 Agent

fullchain-block-audit 聚焦于搜索排序系统中的全链路屏蔽审计这一个职责。它不是”在线问题处理专家”，不是”代码审查助手”。

如果你的团队未来积累了更多 Skill（如 log-analysis、trace-analysis、code-review），它们可以组合成一个更大的 IncidentAnalysisSkill：

这就是 PDF 说的”更强的模型不如更合理的能力组合”。

原则 6：不是表单——必须自主决策和处理异常

“表单”是”填参数 → 固定流程 → 返回结果”，遇到异常就卡住。

fullchain-block-audit 的自主决策点：

  • 如果用户提供的信息不完整（没说清楚要屏蔽哪些渠道），Skill 应该自己判断”默认覆盖所有已知渠道”
  • 如果某个渠道的 forward_info 不可用，Skill 应该建议”放在 Rank 阶段而非 ReRank”
  • 如果 Redis 读取失败，Skill 应该建议降级策略

不足之处：当前版本的异常处理主要靠文字描述，没有给出具体的”if-else 决策树”。可以加强。

原则 7：质量评估——信息充足的情况下，不需要问人

一个好的 Skill 应该：收到明确的需求后，从头跑到尾，不问”接下来做什么？””要填什么参数？””要继续吗？”

fullchain-block-audit 的设计：

  • Audit Checklist 是自主推进的（10 步顺序执行）
  • Channel Inventory 是预定义的（不需要问用户”有哪些渠道”）
  • Implementation Pattern 是自包含的（代码模板 + 注册方式 + Lua 配置）

用户只需要说”我要在新渠道 X 上实现 UP 主屏蔽”，Skill 就应该自己完成所有步骤。

原则 8：脚本工程标准——三层架构

当前缺失：Tool 层还没有实际脚本。这是最大的改进空间。

三、QoderWork create-skill 规范对照

除了 B 站的 Skill 工程化原则，QoderWork 自身也有一套 Skill 创建规范。下面对照分析：

3.1 YAML Frontmatter

规范要求 是否满足 说明
name ≤ 64 字符 是（23 字符）
name 仅含小写字母/数字/连字符 是
description ≤ 1024 字符 是
description 用第三人称 是 “Audit and implement…” 而非 “I can help you…”
description 包含 WHAT + WHEN 是 WHAT = “full-chain content blocking”；WHEN = “Use when…”
description 包含触发词 是 “block”, “filter”, “blacklist”, “search results”, “recommendations”

3.2 正文结构

规范要求 是否满足 说明
SKILL.md < 500 行 是（约 180 行）
有清晰的分节 是 When to Use / Checklist / Inventory / Pattern / Decisions / SOP / Monitoring
术语一致 是 统一用 “blocking”、”filter”、”extractor”、”channel”
无时间敏感信息 是 没有 “如果是在 2025 年之前” 这类表述
引用文件只有一层 是 只引用了 block-upuser-strategy/ 目录

3.3 设计模式使用

QoderWork 定义了 5 种常用模式。fullchain-block-audit 使用了其中的 3 种：

模式 是否使用 具体体现
Template（模板） 是 Implementation Pattern 给出了代码模板
Workflow（工作流） 是 Audit Checklist 10 步流程
Conditional（条件分支） 是 Pipeline Placement 的 4 种场景分别指导
Examples（示例） 否 可改进：加入”实际审计案例”
Feedback Loop（反馈循环） 否 可改进：加入验证脚本和回退逻辑

3.4 反模式检查

反模式 是否犯了 说明
Windows 路径 否 用 scripts/helper.py 格式
选项过多 否 每种场景给了一个默认方案
时间敏感信息 否 无
术语不一致 否 统一使用 “block”、”filter”、”extractor”
模糊的 Skill 名 否 fullchain-block-audit 明确表达了职责

四、Skill 开发 7 步法对照（来自 PDF）

你的 PDF 给出了 Skill 开发的 7 个步骤。回顾 fullchain-block-audit 的开发过程：

步骤 要求 实际做法 评价
1. 定义能力边界 一句话说清目标、输入、输出 目标：全链路屏蔽审计；输入：屏蔽需求；输出：所有渠道的覆盖 清晰
2. 关注点分离 确定性 → 脚本，探索性 → 模型 大部分是模板套用（确定性），只有”判断字段名”需要模型 可加脚本
3. 先建工具再写 Skill 工具就绪 → 写 Skill → 验证 先写了 C++ 算子代码（工具层），再写 Skill 正确
4. 写 Skill Prompt 信息完整，模型能直接执行 提供了完整模板 + 表格 + SOP 信息充足
5. 高档位模型调试 用最强模型跑通全流程 用当前模型生成了完整方案 已验证
6. 部署到执行环境 不改运行时 Skill 安装到了 QoderWork skills 目录 已部署
7. 降档验证 换便宜模型跑，看是否还能工作 未做 待验证

Step 7 是关键质量检验：如果换一个更弱的模型来执行这个 Skill，它能不能照着 Checklist 和模板完成工作？从当前设计看，大部分步骤是”查表 + 套模板”，弱模型应该能处理。但”判断新资源类型的 UP 主 ID 字段名”这一步需要模型读懂 C++ 代码，弱模型可能做不好。

五、Skill 在 AgentOS 中的定位

你的 AI AgentOS 设计方案.pdf 把 Skill 分成三层：

层级 维护者 fullchain-block-audit 属于哪层
Base（基础层） Skill 工程化团队 否——这不是通用基础设施
Domain（领域层） 业务团队 是——搜索推荐团队的垂直能力
Private（私有层） 个人 否——团队可复用

如果按照 AgentOS 的 SKILL.md frontmatter 规范，完整的声明应该是：

当前的 frontmatter 只有 name 和 description（QoderWork 最低要求），如果要在 AgentOS 上注册，需要补充 tier、version、owner、depends 等字段。

六、Skill 生命周期

按照 PDF 描述的生命周期：

当前状态：已开发 + 已在 QoderWork 本地安装。缺失 CI 注册和自动评测环节。

评测维度（4 维度 11 指标）

维度 指标 对这个 Skill 的含义
有效性 成功率 审计后所有渠道确实都有屏蔽覆盖的比例
空跑率 审计完成但没发现任何遗漏的比例
覆盖率 能识别的渠道类型占全部渠道的比例
稳定性 一致性 同一个需求跑两次，结果是否一致
退化率 随代码库变化，审计准确度是否下降
效率 执行延迟 从”开始审计”到”输出报告”的时间
Token 消耗 每次审计消耗的 LLM token 数
调用次数 需要读取多少个文件
用户价值 采纳率 输出的代码和配置被直接使用的比例
修正率 需要人工修改后才能用的比例
负反馈率 用户标记”没用”的比例

七、具体改进建议

基于以上分析，当前 Skill 可以在以下方面改进：

改进 1：加入审计脚本（Tool 层）

这样模型不需要自己读每个 Lua 文件——脚本做确定性工作，模型只做判断（如何处理缺失的渠道）。

改进 2：加入验证脚本

改进 3：补充 Examples 模式

改进 4：补充 AgentOS frontmatter

如果未来要在 AgentOS 上注册，补充完整的依赖声明和元数据。

八、总结

fullchain-block-audit 作为一个 QoderWork 本地 Skill，已经遵循了两套设计原则的大部分要求：

做得好的：能力边界清晰、关注点基本分离、模型无关性较好、术语一致、结构清晰、信息完整。

可改进的：缺少 Tool 层脚本（审计脚本和验证脚本）、缺少具体案例（Examples 模式）、缺少 AgentOS 完整元数据、缺少降档验证。

核心理念回顾你的 PDF 的结语：“Skill 的本质是工程化的能力单元，而不是 Prompt 工程的技巧集合。” 当前版本在”工程化”上还有提升空间——加入脚本、加入验证、加入评测，才能从”一个好的 Prompt”升级为”一个真正的 Skill”。
